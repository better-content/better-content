# Quest Book Flowchart Language

The quest book visual system exists for comprehension, not decoration. Every chapter should read like a flowchart with one consistent direction.

## Direction

- Read every chapter left to right.
- Put the main progression spine on `y:0`.
- Put prerequisite/support branches above or below the spine, then return them to the spine.
- Put exits, terminals, and cross-chapter transitions at the far right.
- Do not use radial, cog, wheel, constellation, or decorative layouts.
- Show dependency lines by default. The chapter is a flowchart, so lines should communicate real progression dependencies.
- Hide lines only for a specific generated node if the line actively harms comprehension.

## Chapter Groups

- Chapter groups describe related work, not reward tier names.
- Group order implies progression and reward intensity.
- Coin payouts remain the mechanical tier signal; group names should not say Copper, Iron, Brass, Gold, or Platinum.
- Current group order: Orientation, Routes and Settlements, Body and Blood, Relics and Rewards, Workshop Spine, Power and Control, Worlds and Threats, Magic and Matter, Space and Intelligence, Late Matter Branches.
- If a content mod can support at least six concrete, non-redundant quest nodes, it should get its own chapter.
- Do not create standalone chapters for libraries, compatibility shims, performance mods, cosmetic-only helpers, or systems with fewer than six meaningful checkpoints.

## Dependency Graph

- Every visible line should represent a real dependency.
- Every chapter should be one connected local dependency component after ignoring cross-chapter prerequisites.
- Cross-chapter dependencies are locking/progression inputs, not the primary way to make a chapter readable.
- A branch may start from an external prerequisite, but it must connect into the local chapter flow.
- Side systems should either rejoin the chapter spine or be intentionally moved to their own chapter.
- The generator emits explicit `hide_dependency_lines:false` and `hide_dependent_lines:false` on each quest node.
- The generator fails if a chapter splits into disconnected local quest components.

## Positioning

- `x` communicates order. Later work always sits farther right.
- `y:0` communicates the required central path.
- Negative `y` communicates alternate inputs, preparation, or source options.
- Positive `y` communicates support systems, side infrastructure, or economy work.
- Branches should be short and rejoin the main path quickly.
- Long-running parallel systems should get their own chapter instead of becoming a tangled local branch.
- `Major Gates` in the Starting Out group is the compact pack-wide overview: it shows the main parallel streams converging into AE2.

## Shape Meanings

Use a small, stable shape vocabulary.

- `rsquare`: normal process/action node.
- `square`: hard manufactured gate, usually a casing or required block-like component.
- `diamond`: decision, permission gate, branch exit, or cross-chapter transition.
- `none`: reference-only documentation node.

Avoid all other shapes unless the generator has a concrete comprehension reason. In particular, do not use `gear` just because a quest is about power.

## Size Meanings

- `size 0.9`: reference-only.
- `size 1.0`: normal process/action node.
- `size >=1.12`: hard gate, casing, decision, or chapter exit.
- `size 1.75`: especially important gate/capstone.
- `icon_scale 1.18`: icon scale for `size 1.75` major targets.
- `min_width:2`: node label must remain readable because it is a step, gate, economy node, source option, or exit.

No chapter should have more than three `size 1.75` emphasis nodes. If everything is large, nothing is important.

## Subtitle Meanings

Subtitles describe flow role, not chapter flavor.

- `Process`: normal required work.
- `Step N`: ordered procedural work.
- `Hard gate`: required manufactured/component checkpoint.
- `Decision gate`: permission or branch decision.
- `Source option`: one possible source feeding the same downstream system.
- `Economy`: coin, trade, Wares, market, or delivery work.
- `Exit`: transition into another chapter/lane.
- `Reference`: guidebook/documentation only.

## Icons

- Use the actual required item or proof item as the icon source whenever possible.
- Machine-casing nodes should require and display the actual casing item.
- Exit nodes should use the item that proves readiness for the next branch.
- Do not choose icons for visual theme if a more informative required item exists.

## Implementation

The system is generated by `tools/generate_expert_quest_book.mjs` through `visualForQuest()` and `importantQuestIds`. Update the generator first, then regenerate chapter SNBT. Do not hand-style generated chapter files.
